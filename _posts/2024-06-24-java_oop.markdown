> ## 객체지향 프로그래밍이란?

 객체를 중심으로 프로그래밍을 짜는 것. 여기서 객체란 기능을 하는 단위를 말합니다.

 예를들면 자동차가 있다고 합시다. 자동차에 들어가는 여러가지 부품들이 있습니다. 핸들, 타이어, 엔진 등등 이러한 부품들이 모두 객체가 되는 것입니다. 이러한 객체들이 모여 자동차를 이루게 됩니다.

 객체지향 프로그래밍이 갖는 장점 중 하나인 유지보수가 쉽다는 점이 이 부분에서 드러나게 됩니다.

 핸들이라는 객체를 어떠한 자동차에서 가져가 쓸 수 있는 것 처럼 한 번 만들어 놓으면 재사용이 용이하고 유지보수 비용이 적게 들어갑니다. 

 <br>

> ## 객체지향 프로그래밍의 특징

 - **추상화** 
   -  객체에서 공통된 속성과 행위를 추출하는 것
   -  실제 세상을 프로그래밍으로 옮길 수 있게, 필요한 기능들만 가져와서 단순화 하는 것
   - 실제를 객체화 하는 것이 아닌 필요한 정보들로 간소화 하는 것을 의미합니다. 지하철의 노선도가 그 예시 중 하나입니다
   - 추상화가 필요한 이유는 추상화는 불필요한 정보는 숨기고 중요한 정보만을 표현함으로써 프로그램을 간단하게 만들어 줍니다

    <br>

   **추상화가 왜 필요한가?**
    - '기아'와 같은 다른 자동차 브랜드가 추가될 수도 있습니다.
    
      이때 추상화로 '자동차'를 구현 해놓으면 다른 곳의 코드를 수정할 필요 없이 추가로 만들 부분만 새로 생성해주면 됩니다.

      <br>

  - **캡슐화** 
    - 데이터를 은닉하고 그 데이터를 접근하는 기능을 노출시키지 않는다는 의미입니다. (***접근 제어자** 활용)


    #### ***접근 제어자**
    <br>
      
      `public : 모두가 접근 가능`

      `protected : 상속 / 같은 패키지 내 클래스에서 접근 가능`

      `private : 본인만 접근 가능`


<br>
  - **상속성**
    - 상속이란 상위개념의 특징을 하위 개념이 물려받는 것을 의미합니다.
    
      하나의 클래스가 가지고 있는 특징 (데이터와 메서드)들을 다른 클래스가 물려주고자 할 때 상속의 특징을 사용합니다.

      - 장점 
        - 재사용으로 인한 코드가 줄어든다.
        - 범용적인 사용이 가능하다.
        - 자료와 메서드의 자유로운 사용 및 추가가 가능하다.

      - 단점
        - 상위 클래스의 변경이 어려워 진다.
        - 불필요한 클래스가 증가할 수 있다.
        - 상속이 잘못 사용될 수 있다.

  - **다형성**
    - 다형성은 다양한 형태로 표현이 가능한 것을 의미합니다.

      - **오버라이딩(Overriding)**
        - 같은 메서드 이름 / 같은 인자 목록 / 상위 클래스의 메서드를 재정의한다. 
        - 상위 클래스 타입의 객체 참조 변수에서 자동으로 하위 클래스가 오버라이딩한 메서드를 호출해 준다. 

      - **오버로딩(Overroding)**
        - 같은 메서드 이름 / 다른 인자 목록 / 다수의 메서드 중복을 정의

        <br>

> ## 객체지향 프로그래밍의 장단점

  - 장점 
    - 클래스 단위로 모듈화시켜서 개발하기 때문에 업무 분담이 편리하고 대규모 소프트웨어 개발에 적합
    - 클래스 단위로 수정이 가능하기 때문에 유지 보수가 편리
    - 클래스를 재사용하거나 상속을 통해 확장함으로써 코드 재사용이 용이

  - 단점
    - 처리속도가 상대적으로 느림
    - 객체의 수가 많아짐에 따라 용량이 커질 수 있음
    - 설계시 많은 시간과 노력이 필요
  
  <br>

 > ## SOLID (객체 지향 설계 원칙)

 객체 지향적으로 설계하기 위해 SOLID 라 불리는 5가지 원칙이 있습니다.

  - 단일 책임 원칙 (SRP, Single Responsibility Principle)
    - 하나의 클래스는 단 하나의 책임만 가져야 한다.
    - 단일 책임 원칙을 지키지 않을 경우 한 책임 변경에 의해 다른 책임과 관련된 코드에 영향이 갈 수 있다.
  - 개방-폐쇠 원칙 (OCP, Open/Closed Principle)
    - 소프트웨어 요소는 확장에는 열려 있으나 변경에는 닫혀 있어야 한다.
    - 기능을 변경하거나 확장할 수 있으면서 기능을 사용하는 코드는 수정하지 않는다.
  - 리스코프 치환 원칙 (LSP, Liskov Substitution Principle)
    - 프로그램 객체는 프로그램의 정확성을 깨뜨리지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야 한다.
    - 상위 타입의 객체를 하위 타입의 객체로 치환해도, 상위 타입을 사용하는 프로그램은 정상적으로 동작해야 한다.
  - 인터페이스 분리 원칙 (ISP, Interface Segregation Principle)
    - 범용 인터페이스 하나보다 클라이언트를 위한 여러 개의 인터페이스로 구성하는 것이 좋다.
    - 인터페이스는 인터페이스를 사용하는 클라이언트를 기준으로 분리해야 한다.
    - 클라이언트가 필요로 하는 인터페이스로 분리함으로써 각 클라이언트가 사용하지 않는 인터페이스에 변경이 있어도 영향을 받지 않도록 만들어야 한다.
  - 의존관계 역전 원칙 (DIP, Dependency Inversion Principle)
    - 추상화에 의존해야지 구체화에 의존하면 안된다.
    - 고수준 모듈은 저수준 모듈의 구현에 의존해서는 안되고 저수준 모듈은 고수준 모듈에서 정의한 추상 타입에 의존해야 한다.
